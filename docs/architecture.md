# Sentinel-Fin — System Architecture

**Version:** 0.1 (pre-implementation design)
**Author:** Kartik
**Last updated:** Week 6, Day 7

---

## 1. System purpose

Sentinel-Fin is an ML-driven fraud detection system that scores banking transactions in near-real-time and correlates the results with security telemetry (authentication logs, device fingerprints, geo-location) to produce prioritized, MITRE ATT&CK-tagged alerts for SOC analysts. Every alert carries SHAP-based feature explanations, supporting RBI model risk management guidance and PMLA reporting requirements (STR filing by the bank's Principal Officer).

The system is built as a capstone demonstration project, not a production deployment. The design is faithful to how such a system would be built at a mid-size Indian bank or NBFC, but operates on synthetic data and runs locally.

---

## 2. Primary user

The primary user is a **Tier-1 SOC analyst** at a mid-size Indian bank or NBFC who:

- Triages 50–200 fraud alerts per shift
- Needs a single pane of glass that combines transaction risk and security context (not two separate dashboards)
- Must justify each escalation with evidence (which features triggered the alert, what auth anomalies coincided, what MITRE technique it maps to)
- Escalates high-confidence alerts to the bank's Principal Officer for STR filing with FIU-IND within the 7-day PMLA window

A secondary user is a **fraud risk modeller / compliance reviewer** who periodically inspects SHAP explanations to confirm the model is not relying on prohibited features (e.g., directly using protected attributes) and to validate that the model still performs within tolerance on recent data (model drift monitoring is out of scope for the capstone but explicitly mentioned as future work).

## 3. Components

The system has **three core components** plus a thin integration layer.

### 3.1 ML Scoring Service

- **Input:** A single transaction event as JSON, containing transaction features (amount, card type, merchant category, time-of-day, device ID, billing/shipping address hash, email domain) and identity features (account age, prior transaction count, behavioural baselines).
- **Output:** A JSON response containing `fraud_probability` (float 0–1), `risk_tier` (LOW / MEDIUM / HIGH / CRITICAL based on tuned thresholds), `top_shap_features` (list of 5 features with signed contributions), `model_version`, and `scoring_latency_ms`.
- **Brief description:** An XGBoost classifier (primary model) ensembled with an Isolation Forest (anomaly score for out-of-distribution behaviour) is wrapped behind a FastAPI endpoint. The XGBoost model is trained on the IEEE-CIS Fraud Detection dataset with class-imbalance handling (`scale_pos_weight`) and PR-AUC as the primary tuning metric. SHAP TreeExplainer generates per-prediction explanations on the fly. The model is loaded once at service start (`joblib`) and held in memory.

### 3.2 Security Telemetry & Correlation Engine

- **Input:** A stream of authentication events as JSON (login attempts: timestamp, user ID, source IP, geo-location, device fingerprint, success/failure) and a stream of transaction events (the same events being scored above). In the capstone, both streams are generated by `stream_simulator.py` from synthetic log files; in production, both would come off a Kafka topic (Week 9 extension).
- **Output:** Correlation events — when patterns of interest are detected, the engine emits a `correlation_alert` JSON object containing the triggering transaction ID, related auth events within the time window, the matched MITRE ATT&CK technique(s), and a correlation confidence score.
- **Brief description:** A Python-based rules engine with two layers. Layer 1 — *deterministic rules* — fires on hard-coded patterns: impossible travel (two successful logins from geographies >500km apart within 10 minutes), brute-force precursor (≥5 failed logins followed by a success within 30 minutes), new-device-then-high-value-transaction (login from never-seen device fingerprint within 5 minutes of a transaction above the 95th percentile of the user's history). Layer 2 — *MITRE mapping* — annotates each fired rule with the relevant ATT&CK technique (T1078 Valid Accounts, T1110 Brute Force, T1566 Phishing as precursor, T1078.004 Cloud Accounts where applicable). Rules are defined declaratively in a single Python module (`integration/correlation_rules.py`) so additional rules can be added without touching the engine itself — this mirrors how production fraud teams build "rule packs" on top of a core engine.

  **Note on SIEM choice:** A production deployment would feed auth logs into a SIEM (Wazuh, Splunk, Elastic) and the correlation engine would consume SIEM alerts rather than raw logs. For this capstone, a SIEM is out of scope — the custom engine reads synthetic auth events directly. This is a deliberate scoping choice (see §7), not an oversight, and is justified by the fact that the *interesting* engineering — fraud-specific correlation rules and MITRE mapping — is the same regardless of which SIEM produces the upstream alerts.

### 3.3 Analyst Dashboard

- **Input:** Combined alerts from the ML scoring service (high-risk transactions) and the correlation engine (security events). Both are persisted to a SQLite database (`alerts.db`) and read by the dashboard.
- **Output:** A web UI for the SOC analyst. Three views: (a) Alert queue, sorted by risk tier and timestamp; (b) Alert detail — per-alert deep dive showing the transaction, the SHAP plot, related auth events, the MITRE technique with link to attack.mitre.org, and a suggested action; (c) Model performance — recent PR-AUC, alert volume, and a "model risk" disclaimer panel referencing RBI guidance.
- **Brief description:** Built with Streamlit. The "suggested action" field for high-confidence alerts reads *"Escalate to Principal Officer for STR review"* — never *"Freeze account"*, because PMLA does not authorise unilateral freezing. Each alert detail view includes a one-click button to export the case as a JSON evidence pack (transaction + features + SHAP + auth events + MITRE mapping) for compliance hand-off.

### 3.4 Integration layer (thin)

- **Stream simulator** — `integration/stream_simulator.py`: replays the synthetic transaction and auth logs at a configurable rate (default 5 events/sec), POSTs to the scoring API, and emits auth events to the correlation engine. Producer/consumer interfaces are stubbed so the transport can be swapped to Kafka in the Week 9 extension without changes to business logic.
- **Alert persistor** — `integration/alert_store.py`: writes scoring outputs above a risk threshold and all correlation alerts to SQLite.

## 4. Data flow — worked example

A single end-to-end trace, the kind of thing the demo video will show.

1. **T+0s** — `stream_simulator.py` emits an auth event: user `U_4421` logs in successfully from IP `103.21.x.x` (geo: Bengaluru) using device fingerprint `dev_a1b2c3`.
2. **T+3s** — Same simulator emits a transaction event: user `U_4421` initiates ₹78,000 transfer to a newly-added payee `BENE_NEW_99`. Event is POSTed to ML Scoring Service.
3. **T+3.05s** — XGBoost returns `fraud_probability = 0.87`, `risk_tier = HIGH`. Top SHAP features: `new_payee_age_days = 0` (large positive contribution), `amount_vs_user_p95 = +2.3σ`, `time_of_day_unusual = True`.
4. **T+3.06s** — Alert persistor writes the scoring result to `alerts.db`.
5. **T+3.5s** — Stream simulator emits a *second* auth event for the same user `U_4421`: login from IP `45.227.x.x` (geo: Lagos, Nigeria) using device fingerprint `dev_x9y8z7`.
6. **T+3.51s** — Correlation engine evaluates: two successful logins, geographic distance ~9,200km, time delta 3.5s → **impossible travel rule fires**. Engine maps to **MITRE T1078 (Valid Accounts)** with precursor hypothesis **T1110 (Brute Force)** flagged for review.
7. **T+3.52s** — Correlation engine queries `alerts.db` for any HIGH-tier scoring within ±5 minutes for user `U_4421`. Finds the ₹78,000 alert from step 4. Emits a **`correlation_alert`** linking transaction and auth chain.
8. **T+3.53s** — Dashboard's auto-refresh picks up the new correlation alert. Analyst sees a single combined card: "**CRITICAL — possible ATO + fraudulent transfer**", with both transaction details and auth chain visible, MITRE T1078 tagged, SHAP plot showing why the model flagged it, and suggested action: *"Escalate to Principal Officer for STR review (PMLA 7-day window)"*.
9. **T+ analyst time** — Analyst clicks "Export evidence pack", downloads `case_4421_20260818.json`, hands off to compliance.

## 5. Tech stack

| Layer | Choice | Why |
|---|---|---|
| ML model | XGBoost + Isolation Forest, SHAP TreeExplainer | XGBoost is the industry default for tabular fraud, SHAP is the gold standard for interpretability under model risk reviews |
| ML serving | FastAPI + uvicorn | Fast, async, auto-OpenAPI docs, recruiter-friendly |
| SIEM | Out of scope (custom correlation engine consumes raw auth events) | Deliberate scoping decision — see §7. A production deployment would place Wazuh / Splunk / Elastic upstream of the correlation engine. |
| Correlation engine | Custom Python (rules + MITRE mapping) | A real fraud team builds custom logic here because off-the-shelf rules can't encode bank-specific patterns |
| Stream transport | File-replay simulator (capstone), Kafka (Week 9) | Deliberate scoping decision — see §7 |
| Storage | SQLite | Zero-config, sufficient for capstone alert volumes (~10K alerts) |
| Dashboard | Streamlit | Fastest path to a credible analyst UI; React would be 5× the work for no demo benefit |
| Deployment | Render free tier (API + dashboard) | Render handles the Python services; everything runs locally for the demo recording |
| Version control | GitHub (public), pinned to profile | — |

## 6. Success metrics

### 6.1 ML model targets

- **Primary:** PR-AUC ≥ 0.65 on holdout (the IEEE-CIS public leaderboard's strong baselines sit ~0.70 PR-AUC; ≥0.65 is honest and defensible)
- **Recall at fixed precision:** ≥0.55 recall at 0.30 precision — meaning when the model says "fraud" it's right 30% of the time, and it catches 55% of all fraud. Low precision is acceptable for fraud because alerts are human-reviewed, not auto-blocked.
- **Scoring latency:** p95 < 100ms on a 4-core laptop, matching typical real-time fraud SLAs
- **Calibration:** Brier score < 0.05 — the predicted probabilities should actually mean something, not just rank

### 6.2 System targets

- **Throughput:** Handle ≥50 events/sec sustained (auth + transaction combined) on a 4-core laptop
- **Alert volume realism:** At 50 events/sec, system should produce 5–15 HIGH/CRITICAL alerts per minute — neither zero nor a flood
- **Dashboard refresh:** New alerts visible within 5 seconds of generation
- **False positive rate at HIGH tier:** ≤30% (acceptable for capstone; production target would be ≤5%)

## 7. Scope — what is explicitly NOT being built

Being explicit about non-goals is a deliberate design discipline. The following are out of scope **by design, not by accident**:

| Non-goal | Why not | Where it would go in production |
|---|---|---|
| SIEM tool integration (Wazuh / Splunk / Elastic) | Setup overhead (~1–2 days) without adding to the *judgment layer* the capstone is showcasing. The custom correlation engine demonstrates the same fraud-specific rule-writing skill, and a real production system would simply place the SIEM upstream of the same engine. | Wazuh or Splunk ingests raw logs and emits structured alerts to a Kafka topic; correlation engine consumes from that topic instead of raw events. |
| Kafka streaming | Time cost (~3–4 days) vs learning ROI — capstone is about the *judgment layer*, not infra plumbing. *Will be added as Week 9 extension.* | Production-grade event bus, replay capability for retraining |
| Real banking API integration | No safe path to integrate with actual UPI/NEFT rails as a student | Bank-specific adapters per channel (UPI, IMPS, card networks) |
| User authentication on the dashboard | A capstone running on synthetic data has no real users to protect | SSO/SAML against the bank's identity provider |
| Model drift monitoring (Evidently AI, custom KS tests) | Would take 3–4 days alone and adds little to the demo | A required production component — flagged in README's "Future work" |
| Adversarial robustness testing | Out of scope for a single-modeller capstone | Red-team exercises against the model |
| Encryption at rest, audit logging, SOC2 controls | Compliance overhead inappropriate for synthetic-data demo | Standard enterprise security baseline |
| Multi-region deployment, HA, DR | Single-laptop demo | Standard cloud architecture |
| Cost-sensitive learning beyond `scale_pos_weight` (e.g., custom loss functions weighted by transaction amount) | Worth doing but adds days to the modelling phase | Genuinely interesting research direction; flagged as future work |

## 9. Risks and mitigations

| Risk | Mitigation |
|---|---|
| ML model PR-AUC below 0.55 | Means feature engineering is the issue. Spend an extra day on the identity-table joins and merchant-category aggregations before tuning XGBoost parameters. |
| Dashboard becomes a time sink | Streamlit's default components are sufficient. Resist the urge to write custom CSS. The story is in the data shown, not the styling. |
| Correlation engine generates too many alerts | Tune the time windows and thresholds based on observed event rates. Document final values in `integration/correlation_engine.py` docstrings. |
| Render free tier kills the API on inactivity | Either accept the cold-start, switch to Railway, or include a screen-recording in the README rather than asking reviewers to wake the service. |
